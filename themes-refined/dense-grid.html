Hexo & VSCode

2022-05-28 · 4 min read · tags: meta, hexo, dev

Setting up a static blog with Hexo seemed straightforward enough on paper, but the real challenge was building a comfortable writing workflow around it. I wanted to write and preview posts from anywhere -- including remote machines -- without having to run a full local dev environment. The answer turned out to be VSCode Remote SSH combined with a Docker-based Hexo setup that could hot-reload content changes on the fly. After more than 25 years carrying around a lightly modified .vimrc, switching to an IDE felt like heresy, but the live markdown preview alone justified the leap.

The core of the setup lives in a docker-compose.yml that mounts the content directory into the container. When you run docker-compose --profile serve up, Hexo watches the source/_posts/ directory for changes and regenerates on save. VSCode connects over SSH, opens the workspace, and suddenly you have full IntelliSense, markdown preview, and integrated terminal access to the container -- all from a browser tab if you want it. The development loop shrinks to something almost trivial: write a sentence, save, glance at the preview pane, keep going.

The best writing environment is the one that gets out of your way. If you have to think about your tools more than your content, something is broken.

Asset management was the one thing that tripped me up early on. Hexo supports post asset folders -- a directory with the same name as your post that holds images and other files -- but the default markdown image syntax doesn't resolve paths correctly. The fix is the hexo-asset-link plugin, which rewrites relative image paths at build time so they point to the right place in the generated output. Once that was in place, dropping a screenshot into the asset folder and referencing it with a standard ![alt](image.png) just worked, no path gymnastics required.

The final piece was deployment. A small deploy.sh script pulls the latest Docker image, runs hexo generate inside the container, and copies the output to a timestamped directory on the host. A symlink gets atomically swapped to point at the new build, which means zero-downtime deploys and trivial rollbacks if something goes wrong. The whole pipeline -- write, save, push, deploy -- takes under thirty seconds from commit to live. It is not the most sophisticated CI/CD in the world, but it has the virtue of being something I actually understand end to end, and that counts for more than most people think.
