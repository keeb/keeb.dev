Hexo & VSCode

2022-05-28

Setting up a static blog with Hexo seemed straightforward enough on paper, but the real challenge was building a comfortable writing workflow around it. I wanted to write and preview posts from anywhere -- including remote machines -- without having to run a full local dev environment. The answer turned out to be VSCode Remote SSH combined with a Docker-based Hexo setup that could hot-reload content changes on the fly. What drew me to this pairing was the seamless feeling of working locally when you are actually connected to a box three time zones away, the latency so low you forget the boundary between machines even exists.

The core of the setup lives in a docker-compose.yml that mounts the content directory into the container. When you run docker-compose --profile serve up, Hexo watches the source/_posts/ directory for changes and regenerates on save. VSCode connects over SSH, opens the workspace, and suddenly you have full IntelliSense, markdown preview, and integrated terminal access to the container -- all from a browser tab if you want it. There is something deeply satisfying about the feedback loop: save a file, glance at the preview pane, and see your prose rendered in the exact theme your readers will see. No mental translation layer, no guessing what the final output will look like.

The best writing environment is the one that gets out of your way. If you have to think about your tools more than your content, something is broken.

Asset management was the one thing that tripped me up early on. Hexo supports post asset folders -- a directory with the same name as your post that holds images and other files -- but the default markdown image syntax doesn't resolve paths correctly. The fix is the hexo-asset-link plugin, which rewrites relative image paths at build time so they point to the right place in the generated output. Once that was in place, dropping a screenshot into the asset folder and referencing it with a standard ![alt](image.png) just worked. No more broken thumbnails, no more wrestling with absolute paths that only resolve in production. It is a small thing, but small frictions compound into abandonment if you let them.

Theme customization deserves its own mention. The Cactus theme ships with sensible defaults, but I wanted something darker and more personal -- something that felt like a terminal session rather than a Medium clone. That meant diving into the EJS templates and Stylus partials, overriding CSS variables for background, accent, and text colors, and adding a few visual touches like the scanline overlay and the ASCII header animation. The theme's architecture made this surprisingly painless: most visual properties are exposed as variables in a single config partial, so you can reshape the entire aesthetic without rewriting layout logic.

The final piece was deployment. A small deploy.sh script pulls the latest Docker image, runs hexo generate inside the container, and copies the output to a timestamped directory on the host. A symlink gets atomically swapped to point at the new build, which means zero-downtime deploys and trivial rollbacks if something goes wrong. The whole pipeline -- write, save, push, deploy -- takes under thirty seconds from commit to live. That speed changes the way you think about publishing. When deploying is nearly free, you stop hoarding drafts and start shipping smaller, more frequent updates. The tooling disappears, and all that remains is the writing itself.
