==[ Article 0x03 ]======================================================

==  I made a keylogger                                                 =
==                                                                     =
==  by keeb                                          2022-06-12        =
==                                                   vol.1 issue #2    =
========================================================================

  [ security research -- understanding input capture ]

There's a particular kind of knowledge you can only get by building the thing yourself. Reading about keyloggers in a textbook tells you what they do. Building one tells you how the entire input pipeline of an operating system actually works -- from the moment a key is pressed on the hardware, through the interrupt handler, into the kernel's input subsystem, and finally up to userspace where your program is sitting there, quietly listening.

I built a keylogger as a learning exercise. Not to deploy it, not to steal credentials, but to understand the attack surface. If you want to defend against something, you need to know exactly how it operates. The best locksmiths know how to pick locks. The best security engineers know how to write malware. That's not a contradiction -- it's a prerequisite.

The implementation itself was straightforward. Hook into the input events, buffer the keystrokes, write them out. What surprised me was how little code it took. The entire thing fit in fewer lines than most people's .bashrc. That's the lesson: the barrier to building offensive tools is absurdly low. The only real defense is understanding the attack deeply enough to detect and prevent it at every layer.

The exercise reinforced something I already believed: you learn more from building than from reading. Every hacker knows this. The documentation is the source code. The classroom is the terminal.
