HEXO & VSCODE

Setting up a proper development workflow for a Hexo blog inside VSCode turned out to be one of those tasks that sounds trivial and then eats your entire afternoon. The goal was simple: write Markdown, preview it live, and deploy without touching a browser or running manual build commands. The reality involved Docker containers, file watchers, and a fair amount of profanity.

The core of the setup is a docker-compose.yml that mounts the source directory into a container running hexo server. VSCode connects to the container workspace, so you get full IntelliSense for the theme's EJS templates and access to the integrated terminal for running Hexo commands directly. Hot reload works out of the box — save a file and the browser tab refreshes within a second.

The best dev environment is one where the distance between thought and published output is as close to zero as possible.

Asset management was the last piece. Hexo's post_asset_folder option creates a directory alongside each post for images and other files. Combined with the hexo-asset-link plugin, you can reference images with plain Markdown syntax and they resolve correctly in both the editor preview and the generated site. No more broken image paths in production while everything looks fine locally.
