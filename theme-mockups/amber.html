Hexo & VSCode

2022-05-28 — keeb

When I set out to build this site I had a few constraints in mind: it had to be statically generated, trivially deployable, and comfortable to write in. I have used Hugo, Jekyll, and a half-dozen bespoke static-site generators over the years, but Hexo hit a sweet spot I did not expect. It is Node-based, fast enough for a personal blog, and the theme ecosystem—while smaller than Hugo's—has some genuinely well-crafted entries. The Cactus theme in particular felt right: minimal, terminal-adjacent, and easy to bend to my will with a few hours of CSS surgery.

The real unlock was the authoring workflow. Every post is a Markdown file in app/web/overlay/writing/, and VSCode with a handful of extensions turns into a surprisingly competent writing environment. Spell-check, Markdown preview, integrated terminal for hexo generate—it all Just Works. I mapped a keybinding to spin up the local Docker preview so I can see rendered output in under two seconds from saving a file.

The deploy pipeline is equally simple: a single shell script that builds the Docker image, generates the static output, and symlinks it into the Nginx document root with a timestamped directory. Rollback is just pointing the symlink at the previous timestamp. No CI service, no build queue, no YAML pipeline definitions—just ./scripts/deploy.sh and a cup of coffee.

The best tools are the ones that disappear. You stop thinking about the tool and start thinking about the work. Hexo and VSCode, together, get close to that ideal for me—a warm terminal, a blinking cursor, and nothing else between thought and published word.
