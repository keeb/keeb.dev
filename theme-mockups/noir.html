Hexo & VSCode

Setting up a static site with Hexo is one of those tasks that sounds straightforward until you actually sit down to do it. The documentation is decent, the ecosystem is mature, but the gap between "hello world" and "this is actually my site" is wider than it looks. The real work is in the configuration, the theme customization, and the deployment pipeline.

VSCode becomes indispensable here. Not because it does anything magical, but because the combination of integrated terminal, markdown preview, and file tree navigation makes the feedback loop tight. You edit a post, glance at the preview, run hexo generate in the terminal below, and see the result. No context switching. No alt-tabbing between five windows.

The theme layer is where most of the time goes. Hexo themes are essentially EJS templates with Stylus or SASS stylesheets, and the Cactus theme in particular has a clean structure that rewards careful reading before modification. Every change to a partial or a layout file cascades through the entire site, so understanding the inheritance model matters.

The best toolchain is the one that disappears. When you stop thinking about the build system and start thinking about the words, you have arrived.

Deployment is the final piece. A simple script that builds the Docker image, generates the static files, and symlinks the output to the serve directory. No CI/CD platform, no webhook chains. Just a shell script and ssh. Sometimes the simplest approach is the most reliable one.
