Terraform with Glitter

// 2025-09-25

Your Terraform is boring because YOU are boring. Let me say that again for the people in the back writing resource "aws_instance" "main" for the eleven-thousandth time like it means something. It doesn't. You named it "main" because you couldn't be bothered to think about what it actually does, who it serves, or why it exists. You've turned one of the most powerful creative mediums in modern engineering into a compliance spreadsheet. Infrastructure-as-code was supposed to be a revolution. You made it a chore.

Here's the thing nobody wants to admit: a Terraform module is a design document. It's architecture rendered in plain text. Every resource block is a decision. Every variable is a contract. Every output is a promise to the next person down the line. So why do we treat it like we're filling out tax forms? When you name a security group "allow_all" you're not just being lazy -- you're actively hostile to anyone who has to read that at 3 AM during an incident. Your naming conventions should tell stories. Your module structure should read like a blueprint with intention behind every wall and window.

I started treating terraform plan as a creative review. Not "does this work" but "does this communicate." Does this plan output tell a story that a half-asleep on-call engineer can follow? Does the diff read like a deliberate change or like someone sneezed on a keyboard? The plan is your audience's first encounter with your ideas. If it reads like garbage, your ideas are garbage -- or at least you've failed to prove otherwise. A good plan output should feel like reading a changelog written by someone who gives a damn.

Modules that read like architecture aren't just aesthetic choices -- they're operational survival. When your infrastructure tells a story, people understand what they're changing before they change it. When your resource names carry meaning, your monitoring dashboards become readable prose instead of cryptographic puzzles. The glitter isn't superficial. The glitter is the difference between infrastructure that humans can reason about and infrastructure that only the person who wrote it can navigate. And that person left the company eight months ago.

Stop writing Terraform like it's a punishment. Start writing it like it's the manifesto for the system you wish existed. Because that's exactly what it is. Every resource block is a declaration of intent. Every module call is an act of composition. You're not configuring servers -- you're building worlds. Act like it.

# --- manifesto/main.tf --- # A system that knows what it is and isn't afraid to say so. module "the_watchtower" { source = "./modules/observation-deck" purpose = "See everything. Miss nothing." alert_channels = ["the-war-room", "night-watch"] retention = "long_memory" } resource "aws_security_group" "the_velvet_rope" { name = "velvet-rope-${var.environment}" description = "You're not on the list." vpc_id = module.the_neighborhood.vpc_id ingress { description = "Trusted friends only" from_port = 443 to_port = 443 protocol = "tcp" cidr_blocks = var.the_inner_circle } tags = { Name = "the-velvet-rope" Manifesto = "Access is a privilege, not a default" Mood = "discerning" } } resource "aws_instance" "cathedral" { ami = data.aws_ami.consecrated_ground.id instance_type = "t3.medium" # humble but sufficient tags = { Name = "cathedral-${var.environment}" Role = "The place where data comes to be transformed" Lineage = "Born from the ashes of main-server-02-final-v3" } }
