AI Is Great at Parsing

ANALYSIS 2025-09-26 — by keeb // STAFF ANALYST

For decades, the act of parsing has been one of the most tedious and thankless tasks in software engineering. Every log format is different. Every network capture has its own quirks. Every legacy protocol dump reads like an ancient dialect nobody alive still speaks fluently. Engineers have spent entire careers writing brittle regular expressions and fragile state machines to extract meaning from the chaos of unstructured data. But something has shifted. Large language models have turned out to be remarkably, almost unsettlingly good at parsing — and the implications are beginning to ripple through every team that touches raw data.

The breakthrough is not that AI can parse faster, though it can. It is that AI can parse flexibly. Consider a network capture where half the packets follow RFC spec and the other half were generated by a firmware bug that mangled the headers. A traditional parser breaks. An AI reads the intent. "These bytes probably represent a timestamp, based on position and range." Sources within a major cloud provider's incident response team, speaking on condition of anonymity, described deploying local models against legacy mainframe transaction logs that had defeated every regex library they threw at them. "We described the output format we wanted and fed it 200 lines of garbage," the source said. "It just... figured it out. It inferred field boundaries from context. We had a clean dataset in forty minutes."

The paradigm shift is subtle but profound: engineers are no longer writing parsers. They are describing what they want extracted. The prompt replaces the grammar definition. A typical interaction might look like this:

# Prompt sent to local model for media file parsing system: "You are a filename parser. Extract structured metadata from media filenames. Return JSON only." user: "Parse these filenames: The.Sopranos.S01E01.Pilot.720p.BluRay.x264.avi Attack on Titan - 01 [1080p].mkv The Matrix (1999) 1080p BluRay.mp4" # Model returns structured JSON for each file: # title, season, episode, year, quality, codec, etc. # No regex. No grammar. No edge cases to handle.

A second anonymous source, this one a security researcher who reverse-engineers malware communication protocols, put it bluntly: "I used to spend a week writing a parser for each new C2 protocol variant. Now I paste a packet dump into an AI and ask it to identify the command structure. It gets it right about 80% of the time on the first try, and the other 20% it gets close enough that one round of correction finishes the job. That is not incremental improvement. That is a category change." The researcher paused. "My parser repository has not had a commit in four months."

The implications extend well beyond convenience. Teams that previously needed dedicated parsing infrastructure — maintained by engineers who understood the arcane specifics of each data format — can now consolidate that work into prompt engineering and validation pipelines. The parser does not need to be maintained, versioned, or debugged in the traditional sense. When the format drifts, the model adapts. When the edge case appears, the model reasons about it rather than crashing into an unhandled exception. This is not the elimination of engineering. It is the relocation of engineering effort from the mechanical to the architectural: defining what correct output looks like, building validation layers, and deciding which data to trust.

"We stopped writing parsers. We started describing outcomes. The model fills in everything between the raw bytes and the clean data."

— Anonymous infrastructure engineer at a Fortune 100 company

BREAKING MCP Is Dead

2026-02-03 — by keeb

Anthropic released MCP as a way to give AI models controlled access to external tools and data sources. The pitch: standardize how AI connects to capabilities and you will get an ecosystem of reusable integrations. Write a tool definition once, use it everywhere. The implicit assumption is that AI needs to be spoonfed — that humans must carefully craft JSON schemas describing what a tool does, what parameters it takes, what it returns.

This is backwards. MCP is a protocol designed by humans to grant AI controlled access to capabilities. It assumes the human is the architect and the AI is the consumer. Every tool definition, every resource schema, every prompt template: written and maintained by humans, for AI to consume. But the trajectory of AI capability suggests that within months, models will be able to discover, negotiate, and invoke capabilities on their own without hand-holding schemas.

The protocol is not wrong. It is already obsolete. The gap between MCP's assumptions and what AI can actually do is widening every week, and no amount of versioning will close it.

BREAKING Building Infrastructure Automation with Claude and Swamp

The answer is a surprisingly coherent story of two systems working together: swamp provides the runtime, CLI, model orchestration, immutable state storage, and validation, while Claude writes code, makes architectural decisions, and invokes the CLI to bring it all together.

Swamp is an AI-native automation framework. It provides a model system for defining typed resources, workflow orchestration, expression evaluation, and persistent state storage. Claude uses the CLI to create models, run methods, and store infrastructure state. Claude writes the TypeScript; swamp compiles, validates, and executes it.

The result is infrastructure-as-conversation rather than infrastructure-as-code. The model asks questions, proposes structures, validates assumptions, and builds a living map of the cluster — all without a single Terraform file in sight.
